## Overview

1. 1차 산출물 → 2차 산출물 가공
2. 2차 산출물에 대한 검색

이 두 단계 이후의 작업은 이 문서의 범위가 아니다.

## 핵심 철학

이 문서 전반의 설계 원칙은 다음과 같다.

> 지도를 그릴 때는 정답을 찾지 말고 오답을 계속 배제하는 방법으로 디자인해야 한다.

이 철학은 **검색 단계**에 적용된다. 2차 가공 단계는 schema.md에 따라 정직하게 분해/재조립하되, LLM이 제안한 재배치가 schema와 어긋날 수 있음을 전제한다. 이 경우에도 즉시 실패로 보지 않고, 사람이 확인할 수 있는 2차 산출물 후보 또는 수정 요청으로 노출한다. 철학은 검색 단계에서 본격적으로 작동한다.

이유는 단순하다. 2차 가공은 사람이 직접 보고 수정 요청을 허락/거절하는 단계라 정답/오답 모델이 필요 없다. 반면 검색은 자동화가 답을 내야 하는 단계이고, "정답을 한 번에 고르는" 방식은 환각과 누락을 만든다. 후보를 깔고 오답을 배제하는 방식이 더 안전하다.

# Part 1. 2차 가공 산출물 생성

## 1.1 입력과 출력

- 입력: 1차 산출물 (`raw/sources/*.md`, raw 파일별 markdown 중간 표현)
- 출력: 2차 산출물 (프로젝트 지식 구조 단위로 재배치된 파일 시스템)
- 안내자: 프로젝트별 `schema.md`

## 1.2 분해와 재조립

1차 산출물 하나는 `raw/sources/*.md` 파일 단위이므로, 그 안에 여러 의미 단위가 섞여 있다.

예를 들어 `raw/sources/instance_server_design.md` (1차 산출물) 안에는 다음이 섞여 있을 수 있다.

- 인스턴스 서버 구조
- 던전 A 입장 조건
- 던전 A 보상
- 던전 B 몬스터 스폰

2차 가공은 이 1차 산출물을 의미 단위로 분해하고, schema.md가 정의한 위치로 재배치한다.

```text
raw/sources/
  instance_server_design.md

db/
  instance_server/
    server_structure.md
    dungeon/
      dungeon_a/
        overview.md
        entry_rules.md
        rewards.md
      dungeon_b/
        spawn_rules.md
```

## 1.3 schema.md의 역할

schema.md는 다음을 정의한다.

- 프로젝트의 지식 저장소 폴더 구조
- 어떤 정보가 어떤 파일에 저장되는지
- 1차 산출물의 내용을 어떻게 분해할지에 대한 안내
- 기존 2차 산출물과 신규 내용을 비교할 때의 기준

schema.md는 프로젝트마다 고유하다. 동일한 시스템이 여러 프로젝트에 쓰일 수 있도록, 분해 규칙은 schema.md에 외부화된다.

schema.md는 강제 스키마 validator라기보다 **가공 기준과 검토 기준**이다. LLM이 schema와 다른 위치를 제안할 수 있으므로, 시스템은 그런 제안을 자동으로 정답 처리하지 않고 사람이 판단할 수 있게 노출해야 한다. 사람이 허락하면 schema 자체가 낡았거나 예외가 필요한 것으로 볼 수 있고, 거절하면 폐기/보류/반례로 기록해 다음 가공에서 같은 오류를 줄인다.

## 1.4 출처 추적

1차 산출물(`raw/sources/*.md`)의 내용은 2차 가공 후 여러 파일로 흩어진다. 따라서 모든 2차 산출물 파일은 출처 추적 정보를 가진다.

```md
# 던전 A 입장 조건

- 최소 레벨: 30
- 파티 인원: 1~4명
- PvP: 불가

## Sources

- instance_server_design.docx > section 3.2
- dungeon_balance.xlsx > DungeonA!B12:E18
```

출처 추적은 다음을 가능하게 한다.

- 수정 요청 검토 시 근거 제시
- 원본 문서 변경 시 영향받는 2차 산출물 식별
- 검색 결과에 대한 근거 출력
- 사람이 신뢰할 수 있는 검토 흐름

## 1.5 수정 요청 처리

2차 가공은 기존 DB에 새로운 내용을 추가하거나 기존 내용을 수정한다. 시스템은 자동으로 기존 내용을 덮어쓰지 않는다. 대신 사용자에게 **수정 요청**을 제시하고, 사용자가 이를 허락하거나 거절한다.

수정 요청이 발생하는 경우는 다음을 포함한다.

- 같은 속성값에 대해 새로운 값이 들어올 때
- 규칙이 기존 규칙과 다르게 들어올 때
- 용어 정의가 기존과 다르게 들어올 때
- 테스트 기대값이 변경될 때
- 기존 문서 내용을 수정해야 할 때
- 기존 DB 구조와 다른 위치에 신규 내용이 배치되어야 할 때

시스템은 수정이 필요한 위치, 기존 내용, 신규 내용, 출처를 사용자에게 보여준다. 사용자는 다음 중 하나를 선택한다.

- **허락**: 신규 내용으로 반영한다.
- **병합**: 기존 내용과 신규 내용을 합쳐 반영한다 (사용자가 직접 편집).
- **거절**: 신규 내용을 반영하지 않는다.

사용자가 거절을 선택할 경우, 시스템은 거절된 신규 내용을 어떻게 처리할지 다시 사용자에게 묻는다.

- **폐기**: 신규 내용을 버린다. 다음 가공에서 동일한 1차 산출물 구간이 다시 들어와도 같은 거절이 반복되지 않도록 거절 사유를 기록한다.
- **보류**: 신규 내용을 별도 영역(`pending/`)에 보관한다. 추후 사람이 다시 검토할 수 있다.
- **반례 등록**: 신규 내용을 "기존 내용이 맞다"는 근거로 사용한다. 출처 추적에 반례로 기록되어, 다음에 동일/유사 신규 내용이 들어오면 시스템이 미리 경고한다.

거절 처리 방식 역시 출처 추적에 함께 기록된다. 이는 "정답을 자동으로 만들지 않는다"는 철학의 가공 단계 표현이다.

## 1.6 local git 연동과 히스토리 관리

2차 산출물 디렉토리는 **local git 저장소**로 관리된다. 시스템이 외부 git 서비스(GitHub 등)에 의존하지 않고도 모든 변경 이력을 추적할 수 있도록, 가공 결과는 항상 로컬에서 commit된다.

- 2차 산출물 루트는 git init된 디렉토리이다.
- 시스템은 가공 작업의 각 단위(수정 요청 허락/병합/거절, 신규 추가, 분해/재배치)를 자동으로 commit한다.
- commit 메시지에는 처리한 수정 요청의 종류, 영향받은 파일, 출처(1차 산출물 위치)가 포함된다.
- 사용자는 언제든 `git log`, `git diff`, `git checkout`을 통해 변경 이력을 조회하고, 잘못된 가공을 되돌릴 수 있다.

이 구조는 다음을 보장한다.

- **히스토리**: 누가/언제/어떤 수정 요청을 어떻게 처리했는지가 git 이력에 남는다.
- **되돌리기**: 잘못 허락된 수정도 commit 단위로 revert 가능하다.
- **외부 의존 없음**: local git만으로 충분하며, 원격 동기화는 선택 사항이다.
- **검토 흐름과의 일관성**: 사용자는 익숙한 git 도구로 가공 결과를 검토한다.

## 1.7 가공 흐름 요약

```text
raw 파일
   ↓ (1차 가공: 파일별 markdown 변환)
1차 산출물 (raw/sources/*.md)
   ↓ (의미 단위 분해)
의미 블록들
   ↓ (schema.md에 따른 재배치)
2차 산출물 후보 (지식 구조 단위)
   ↓ (기존 내용과 다르거나 schema와 어긋난 배치로 보이면)
사용자에게 수정 요청 제시
   ↓ (허락 / 병합 / 거절)
   ↓ (거절이면: 폐기 / 보류 / 반례 등록 중 선택)
확정된 2차 산출물 + 출처 추적
   ↓
local git commit (히스토리 기록)
```

# Part 2. 2차 가공 산출물 검색

## 2.1 검색이 풀어야 할 문제

2차 산출물은 사람이 읽기 위한 지식 저장소가 아니라, 자동화된 검증/추론에 쓰일 자산이다. 따라서 검색은 단순한 키워드 매칭이 아니라, 다양한 목적에 따라 **다른 방식으로 후보 공간을 좁히는** 과정이다.

검색의 핵심은 다음과 같다.

> 정답을 한 번에 고르려 하지 말고, 명백한 오답을 배제한 뒤 남는 것을 답으로 본다.

## 2.2 검색의 기본 모델

검색은 다음 흐름으로 동작한다.

```text
질문
  ↓
[1] 질문 유형 판정
  ↓
[2] 유형 기반 배제: 이 유형이면 무조건 배제되는 문서 집합 적용
  ↓
[3] 줄어든 후보 공간을 시작 상태로 탐색
  ↓
잔존 결과 제시
```

배제는 **검색 진입 시점에 한 번** 적용된다. 탐색이 시작된 이후에는 배제가 추가로 일어나지 않는다. 이는 다음을 위한 결정이다.

- 단순함과 추적 가능성
- "어떤 문서가 왜 배제됐는가"를 사람이 검증하기 쉬움
- 배제 지도가 검색 시작 상태를 결정한다는 명확한 멘탈 모델

## 2.3 질문 유형

검색은 질문이 어떤 유형인지 먼저 판정한다. 유형은 검색의 목적이며, 같은 2차 산출물에 대해서도 유형별로 의미 있는 후보 공간이 다르다.

질문 유형은 프로젝트마다 정의될 수 있으며, 시스템은 이를 외부 파일로 관리한다.

```text
question_types/
  condition_based_test.md
  combination_explosion.md
  regression_test.md
  test_gap.md
  risk_recommendation.md
  change_impact.md
  design_defect.md
  policy_violation.md
  balance_simulation.md
  user_experience_warning.md
  game_screen_understanding.md
  next_action_suggestion.md
```

각 유형 파일은 다음을 정의한다.

- 유형의 의미와 적용 조건
- 이 유형의 입력 형태
- 이 유형이 기대하는 출력 형태
- 이 유형에서 항상 배제되는 문서 집합 (다음 절 참조)

## 2.4 질문 유형 기반 배제

배제는 **질문 유형 기반의 정적 배제** 단일 축으로 구성된다.

질문이 특정 유형이면 무조건 배제되는 문서가 정해져 있다. 이는 검색이 시작되기 전에 적용되는 사전 배제이다.

```text
exclusions/
  by_question_type/
    regression_test.md
    policy_violation.md
    test_gap.md
    ...
```

각 파일은 다음을 정의한다.

- 이 유형에서 배제될 폴더/파일/노드 집합
- 배제의 근거 (왜 이 유형에서 이 문서가 무관한가)
- 출처 (사람의 판단인가, 도메인 정책인가, 자동 도출인가)

예: `policy_violation.md`

```md
# 정책 위반 탐지 유형의 배제 대상

## 배제

- db/instance_server/server_structure.md
  근거: 서버 구조는 정책이 아닌 인프라 설명. 정책 위반 판정과 무관.
- db/**/spawn_rules.md
  근거: 스폰 규칙은 게임 정책이 아닌 운영 데이터. 정책 위반과 분리.

## 출처

- 사람 검토 (2026-04-12)
- 도메인 정책: 정책 = Policy/Intent Layer 한정
```

### 동적(내용 기반) 배제를 두지 않는 이유

질문 본문에 등장한 키워드/개념에 따라 후보를 추가로 배제하는 "내용 기반 동적 배제"는 채택하지 않는다. 이유는 다음과 같다.

- 사람이 그 규칙을 사전에 검토·통제하기 어렵다. 매핑(동의어/상하위/도메인 변환)이 끼어들면 같은 질문이라도 시점·표현에 따라 다른 후보 공간이 만들어진다.
- 잘못 배제된 문서를 사람이 역추적하기 어렵다. 동적 매핑은 결과만 보여주고 사용자가 매핑 자체를 손볼 단위가 명확하지 않다.
- 검색의 시작 상태가 사람의 명시적 판단(질문 유형 + 사전 배제 파일)으로만 결정되어야, 배제 지도 전체를 사람이 git으로 관리할 수 있다.

질문 본문에 따른 좁히기는 배제가 아니라 **탐색 단계의 우선순위**로만 다룬다 (탐색 알고리즘은 이 문서 범위 밖).

## 2.5 배제의 적용 흐름

검색이 시작될 때 배제는 다음 순서로 적용된다.

```text
초기 후보 공간 = 전체 2차 산출물

[1] 질문 유형 판정 → 유형 T 결정
[2] exclusions/by_question_type/T.md 적용 → 후보 공간 축소
[3] 줄어든 후보 공간이 검색의 시작 상태가 된다
```

이 시점부터 본격적인 탐색이 시작된다. 탐색의 구체적 알고리즘은 검색 목적에 따라 다르며, 이 문서에서는 **줄어든 후보 공간 위에서 동작한다**는 사실까지만 다룬다.

## 2.6 배제 기록의 누적과 재사용

배제는 일회성 행위가 아니다. 한 번 사람이 검토하고 합의한 배제는 다음 검색에서 그대로 재사용되어야 한다. 따라서 배제 자체가 자산이 된다.

배제는 세 가지 추상화 레벨로 누적된다.

### Level 1. Instance Exclusion

특정 검색 1회에서 발생한 구체적 배제. 로그성 기록.

```text
exclusions/instances/2026-05/
  query_0421_001.md
```

### Level 2. Pattern Exclusion

Level 1이 일정 빈도로 반복되면 패턴으로 승격된다. 패턴은 질문 유형 축에 묶여 저장된다.

```text
exclusions/by_question_type/
```

### Level 3. Axiom Exclusion

도메인적으로 본질적이고 안정적인 배제. 사람 검토를 거쳐 axiom으로 승격된다.

```text
exclusions/axioms/
  spawn_rules_vs_rewards.md
  safezone_vs_pvp_only.md
```

승격은 자동이 아니다. 빈도 신호 + 사람의 명시적 승인이 결합된다. 자동 승격은 잘못된 배제를 굳히는 위험이 있다.

## 2.7 배제 지도의 파일 시스템 구조

배제 지도는 schema.md와 동일한 결로 사람이 읽고 수정 가능한 파일 시스템에 둔다.

```text
exclusions/
  exclusion_schema.md          # 배제 좌표계와 적용 순서 정의
  promotion_rules.md           # Level 1 → 2 → 3 승격 기준

  axioms/                      # Level 3
    spawn_rules_vs_rewards.md
    safezone_vs_pvp_only.md

  by_question_type/            # Level 2 (유형 축)
    condition_based_test.md
    regression_test.md
    policy_violation.md
    test_gap.md
    ...

  instances/                   # Level 1
    2026-05/
      ...
```

이 구조는 다음을 보장한다.

- 사람이 git처럼 검토/수정/되돌리기 가능
- 잘못된 배제 발견 시 해당 파일만 고치면 다음 검색부터 반영
- 변경 이력 관리는 2차 가공의 local git 흐름을 그대로 재사용

## 2.8 배제의 무효화와 신선도

배제 지도는 영원히 누적되지 않는다. 잘못된 배제, 변경된 도메인, 폐기된 정책에 대응해야 한다.

- **출처 의존성**: 배제 규칙은 자신이 의존하는 출처(문서/정책/사람의 판단)를 명시한다. 출처가 변경되면 의존하는 배제 규칙은 자동으로 "재검토 필요" 상태로 전환된다.
- **신선도**: `last_validated_at`이 오래된 axiom은 사람에게 재확인 요청이 발생한다.
- **반례 발견**: 배제된 후보가 다른 경로(사람 직접 지정, 다른 유형의 검색 결과)에서 정답으로 채택되면 해당 배제 규칙이 무효화 후보로 표시된다.
- **명시적 폐기**: 사람이 axiom 또는 패턴을 archived 처리할 수 있다.

이 네 가지 메커니즘은 배제 지도가 시간에 따라 자기 정정되도록 만든다.

## 2.9 검색 결과에 첨부되는 배제 추적

검색 결과는 답만 제시하지 않는다. **무엇이 왜 배제되었는지**도 함께 제시한다.

```text
질문: Fireball을 SafeZone에서 발사할 때 어떻게 되는가?

판정된 유형: 조건 기반 가상 테스트
적용된 배제:
  - 유형 기반: 47개 문서 중 21개 제거
    근거: condition_based_test.md

탐색 시작 후보: 26개

결과: 1개
  → "SafeZone 내 공격성 스킬 차단"

근거: instance_server_design.docx > section 3.2
```

배제 추적은 다음을 가능하게 한다.

- 사람이 검색 결과를 신뢰/검증
- 잘못된 배제를 역추적해 수정
- 배제 지도의 품질을 측정

## 2.10 잔존 0개의 의미

배제 후 잔존이 0개인 경우는 일반 검색에서는 단순히 "결과 없음"이지만, 배제 기반 검색에서는 의미 있는 신호이다.

- 조건 기반 테스트에서 잔존 0 = 명세 공백
- 정책 위반 탐지에서 잔존 0 = 위반 없음 (긍정적 신호)
- 회귀 테스트에서 잔존 0 = 영향 없음
- 테스트 공백 탐지에서 잔존 0 = 공백 없음

따라서 잔존 0은 정상 결과로 취급되며, 그 의미는 질문 유형별로 정의된다. 이 정의 역시 `question_types/<type>.md`에 포함된다.

# Summary

## 2차 가공 산출물 생성

- 1차 산출물은 `raw/sources/*.md`에 저장된 raw 파일별 markdown 중간 표현이다.
- 1차 산출물을 의미 단위로 분해해 schema.md가 정의한 위치로 재배치한다.
- schema와 어긋난 재배치 제안은 즉시 실패가 아니라 사람이 검토할 후보/수정 요청으로 다룬다.
- 모든 2차 산출물은 출처 추적을 가진다.
- 기존 내용과 다른 신규 내용은 자동으로 반영되지 않는다. 시스템은 사용자에게 수정 요청을 제시하고, 사용자가 허락/병합/거절을 선택한다.
- 사용자가 거절하면 시스템은 거절된 내용을 어떻게 처리할지(폐기/보류/반례 등록) 다시 묻는다.
- 2차 산출물 디렉토리는 local git으로 관리되어, 가공의 모든 단계가 commit으로 기록되고 되돌릴 수 있다.

## 2차 가공 산출물 검색

- 검색은 정답을 한 번에 고르지 않고, 오답을 배제한 뒤 남는 것을 답으로 본다.
- 배제는 검색 시작 시점에 한 번 적용된다. (탐색 도중 추가 배제 없음)
- 배제는 **질문 유형 기반의 정적 배제 단일 축**으로 구성된다. 사람이 통제할 수 없는 동적(내용 기반) 배제는 두지 않는다.
- 배제된 문서를 제외한 좁아진 후보 공간이 탐색의 시작 상태이다.
- 배제 기록은 Instance / Pattern / Axiom 세 단계로 누적되며, 승격은 자동이 아닌 빈도 + 사람 승인의 결합이다.
- 배제 지도는 사람이 git처럼 읽고 수정 가능한 파일 시스템으로 관리된다.
- 배제는 출처 의존성, 신선도, 반례 발견, 명시적 폐기 메커니즘을 통해 시간에 따라 자기 정정된다.
- 검색 결과에는 배제 추적이 함께 제시되어, 사람이 결과를 검증하고 배제 지도를 개선할 수 있다.
