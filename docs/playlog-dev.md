
## Phase I. 플레이 녹화 영상·입력 로그 자산화 — Stage 0 변환 도입

**배경**: "플레이 녹화 영상"과 "입력 로그", "입력 목적", "영상에서 발생한 사건"을 받아 (키입력 전후 스크린샷, 키입력, 키입력 목적, 영상에서 나온 현상) 4요소를 지식 자산화하고 싶다는 요구. 기존 파이프라인(Step 1 검색, Stage 1/2 추출, Step 3 그래프 쿼리)은 전부 `db/*.md`의 **텍스트 본문**이 검색·추출 가능하다는 전제 위에 있다(`interchange-spec.md` §2 — 본문은 verbatim source_text). 스크린샷 이미지를 md에 경로/base64로 그대로 넣는 방식은 이 전제를 깨뜨린다 — 토큰 검색은 이미지 바이트/경로 문자열만 매칭하고, Stage 1/2 텍스트 LLM은 이미지를 읽지 못해 사실상 검색·추출 불가능한 죽은 자산이 된다.

**검토한 이미지 표현 방식과 선택**:

| 방식 | 검토 결과 |
|---|---|
| 서술형 텍스트로 변환 | **채택**. 멀티모달 LLM이 스크린샷을 판독해 "체력 40%, 몬스터 그룹 3마리 근접" 같은 자연어 서술로 변환. 기존 토큰/벡터 검색과 Stage 1/2 triple 추출에 그대로 재사용 가능 — 파이프라인 전제를 그대로 만족시킨다. |
| SVG 코드로 저장 | **기각**. 스크린샷은 사진적 정보(전투 이펙트, UI 배치 등)가 많아 벡터 도형 코드로 무손실 변환이 애초에 불가능하고, 시도해도 검색·추출에 쓸 수 있는 의미 있는 텍스트가 나오지 않는다. |
| ASCII 아트(`l----`류) | **채택**. 정보 밀도가 낮지만 화면 이해의 도움됨. |

→ 이미지는 **경로도, 인라인 데이터도 md 본문에 남기지 않는다.** 원본 스크린샷/영상 파일은 별도 보관하지 않고, 본문에 이미지 참조 마크업을 넣지 않는 것이 원칙이다.

**Stage 0 (신규) — 로그/영상 → 서술형 문서 변환**:

기존 ingest는 Stage 1(섹션 분해)이 "이미 사람이 쓴 위키 문서"를 원본으로 가정한다. 플레이 로그는 이 전제를 만족하지 못하므로 Stage 1 앞에 새 변환 단계가 필요하다.

- **입력**: (키입력 전 스크린샷, 키입력 후 스크린샷, 키입력값, 키입력 목적, 영상에서 발생한 사건) 레코드 1건.
- **처리**: 멀티모달 LLM 호출 1회로 전/후 스크린샷 쌍을 판독해 "입력 전 상태 / 입력 / 목적 / 입력 후 상태 및 발생 사건"을 각각 자연어 문단으로 서술한 `db/*.md` 문서를 생성한다. 키입력 목적은 이미 로그에 값이 있으므로 LLM이 지어내지 않고 그대로 인용한다.
- **출력 문서 형식** (기존 `interchange-spec.md` §2-2 양식을 따름):
  ```md
  ---
  title: 스킬A 사용 시퀀스 #142
  status: draft
  sources:
    - 
  ---

  ## 입력 전 상태
  플레이어 체력 40%, 몬스터 그룹 3마리 근접 중.

  ## 입력
  키: Q (스킬A 발동)
  목적: 광역 데미지로 몬스터 그룹 정리

  ## 입력 후 상태 / 발생 사건
  스킬A 시전 후 넉백 발생, 몬스터 2마리 사망, 1마리는 넉백 범위 밖으로 회피함.
  ```
  이후 이 문서는 기존 Stage 1/2가 그대로 처리한다 — 예를 들어 `스킬A --HAS_EFFECT--> 넉백`, `스킬A --USED_FOR--> 광역 정리` 같은 triple이 자연스럽게 추출된다.
- **그래프 프리픽스**: Phase H의 컨벤션에 따라 이 도메인의 그래프는 `playlog_` 프리픽스를 쓴다 (예: `playlog_skill_effect`, `playlog_move_server`).
- **범위 제한**: 이번 계획에서는 Stage 0의 존재와 입출력 형식만 정의한다. 스크린샷 판독에 쓸 멀티모달 모델 선정, 로그 수집 포맷(입력 목적을 사람이 라벨링하는 UI/파이프라인), 대량 로그 배치 처리 방식은 별도 설계 문서(예: `docs/playlog-asset-plan.md`)에서 다룬다 — 이 문서는 "왜 서술형 텍스트여야 하는가"와 "기존 파이프라인과의 접합점"까지만 확정한다.

→ 검증(별도 설계 문서에서 구체화): Stage 0 변환 결과 문서가 Stage 1/2를 통과해 의미 있는 triple을 만들어내는지 샘플 케이스로 확인. 서술형 문서에 원본 스크린샷 경로/데이터가 섞여 들어가지 않는지(위 원칙 위반 여부) 리뷰.

## 마무리

→ `npm run typecheck && npm run test:mocks` + Rust 테스트 통과. 실제 앱 E2E:
1. 쿨타임 언급 문서 여러 개 ingest → 값 노드가 안 생기고 `HAS_COOLDOWN` 계열 relation type이 생기는지 확인.
2. Chat에서 열거형 질문("쿨타임 있는 스킬 전부") → relation 쿼리 기반 나열 확인.
3. Chat에서 개체 중심 질문("스킬A 효과는?") → entity node 쿼리 기반 컨텍스트 확인.
4. **Chat에서 관계 질문("스킬A와 스킬B는 무슨 관계야?") → path 쿼리 기반 경로 나열 확인.**
5. Chat에서 위키와 무관한 질문 → 검색 0건 + Step 3 스킵 + "관련 문서를 찾지 못했다" 답변 확인.
6. Knowledge 탭에서 엔티티 선택 → 여러 그래프에 걸친 병합 뷰 렌더링, hop 조절 확인.

구현 후 `docs/answer-guide.md`를 갱신한다: Step 1(임계값 기반으로 변경), Step 2(제거), Step 3(relation/entity/path 쿼리로 교체).
