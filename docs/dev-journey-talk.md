# 게임 QA를 위한 LLM 지식베이스 개발(고민)기

> 발표 초안. 각 고민은 "문제 → 시도(실패 포함) → 지금의 답 → 배운 점" 구조로 정리했다.
> 고민 1~9는 원안 그대로, 고민 10~14는 개발 이력에서 발굴해 추가한 것이다.

## 발표 흐름 제안 (3막 구성)

| 막 | 주제 | 고민 |
|----|------|------|
| 1막. 표현 | 게임 지식을 어떻게 담을 것인가 | 1, 7, 11, 13, 4 |
| 2막. 신뢰 | 담긴 지식을 어떻게 믿을 것인가 | 2, 6, 15, 5, 10, 12 |
| 3막. 소비 | 담긴 지식으로 어떻게 답할 것인가 | 8, 9, 3, 14 |

(순서는 자유롭게 재배치 가능. 아래 본문은 원안 번호 순서를 유지한다.)

## 고민 1. 어떻게 게임을 LLM에게 이해시킬 것인가 — 지식 그래프로 온톨로지를 어떻게 표현할 것인가

**슬라이드 헤드라인: "온톨로지는 설계 산출물이 아니라 축적의 부산물이었다"**

### 문제
- 게임은 자연어 문서(기획서·패치노트)와 구조화 데이터(스탯·수식)와 암묵지(QA 노하우)가 섞인 도메인이다. LLM에게 "이 게임"을 이해시키려면 무엇을 어떤 형태로 넣어야 하나.
- 교과서적 답은 "온톨로지를 설계하라"인데, 게임마다 도메인이 다르고(스킬/몬스터/퀘스트/UI/경제) 개발 중에는 스키마 자체가 계속 바뀐다. 완성된 온톨로지를 먼저 그리는 것은 불가능했다.

### 시도와 결론
1. **이중 표현을 선택했다 — 문서 + 그래프.**
   - 문서(위키 페이지): 원문을 verbatim으로 보존. 맥락과 근거의 저장소. 사람도 읽는다.
   - 그래프(assertion): 문서에서 추출한 `주어 --술어--> 목적어` triple. 관계 탐색과 다중 홉 추론의 저장소.
   - 핵심 연결고리는 **evidence** — 모든 triple은 자신이 추출된 문서 섹션의 원문 인용(quote)을 근거로 달고 있다. 그래프는 문서 위에 떠 있는 인덱스이지, 독립된 제2의 진실이 아니다.
2. **온톨로지를 미리 그리는 대신, "생성 규칙"만 고정했다.**
   - 사람이 고정하는 것: 엔티티 타입 통제 어휘(`ENTITY_TYPES`), predicate 표기(`UPPER_SNAKE_CASE`), 그래프 명명 규칙(`{목적}_{주어타입}_{행위}_{목적어타입}` — 예: `combat_character_attacks_enemy`, `quest_npc_gives_item`).
   - LLM이 채우는 것: 구체적인 그래프·relation type. ingest 중 기존 카탈로그에 맞는 그래프가 없으면 LLM이 규칙에 맞는 새 그래프를 제안하고, 기존 그래프의 목적 접두사 목록을 프롬프트에 주입해 재사용을 유도한다.
3. **정책 파일 → 코드 인라인으로 진화.** 초기에는 `graph-policy.json`이라는 별도 정책 파일로 온톨로지 규칙을 관리했지만, 결국 규칙은 ingest 코드의 프롬프트+검증 로직으로, 카탈로그는 DB 테이블(`graphs`/`relation_types`)로 인라인됐다. "정책을 데이터로 두면 코드와 어긋난다"는 교훈.

### 배운 점
- 온톨로지의 품질은 설계 회의가 아니라 **명명 규칙 + 기계 검증 + 재사용 유도**에서 나왔다. 사람은 형식을 통제하고, 내용은 데이터가 쌓이면서 스스로 모양을 갖춘다.

## 고민 2. 한 번 들어온 지식의 정확성은 누가 체크하며, 유효 기간은 어떻게 측정하나

**슬라이드 헤드라인: "기계는 형식과 근거를, 사람은 의미를 — 유효기간은 달력이 아니라 모순이 알려준다"**

### 문제
- LLM이 추출한 지식을 무검증으로 쌓으면 오염되고, 전부 사람이 검수하면 병목이다. 검수 책임의 분계선을 어디에 긋나.
- "이 지식은 언제까지 유효한가"는 더 어렵다. 패치 하나로 어제의 진실이 오늘의 거짓이 된다. TTL(유효기간 필드)을 달아볼까?

### 지금의 답 — 3단 방어선
1. **기계 검증 (형식·근거)**: 추출된 모든 triple은 근거 quote가 실제 섹션 본문의 exact substring이어야 한다. 엔티티 타입이 통제 어휘를 벗어나거나, 그래프 이름이 규칙에 안 맞으면 그 chunk 전체가 `VALIDATION_FAILED`로 거부된다. LLM의 "그럴듯한 창작"이 형식 검증을 통과하지 못하게.
2. **충돌 검출 (자동 격리)**: 같은 섹션에 다른 내용이 들어오면 자동 반영하지 않고 modification proposal로 사람에게 올린다. 그래프 쪽은 카디널리티 제약("이 관계는 목적어가 하나여야 함")을 위반하는 두 번째 값이 오면 기존 것을 덮지 않고 `status: review`로 격리한다.
3. **사람의 최종 결정**: Review 화면에서 Approve / Merge(수동 편집) / Reject. 사람은 "이 diff가 맞는 변경인가"라는 의미 판단만 한다.

### 유효 기간에 대한 결론
- TTL 방식은 기각. 게임 지식은 "n개월 뒤 만료"가 아니라 **다음 패치가 언제 그것을 뒤집을지 모르는** 지식이다. 만료 시각은 예측 불가능하다.
- 대신 **모순 주도(contradiction-driven) 만료**: 새 데이터가 들어와 기존과 충돌하는 순간이 곧 낡음이 감지되는 순간이다. 섹션 단위 reconcile과 카디널리티 충돌이 그 감지기다.
- "그땐 맞았던" 상태는 삭제되지 않고 git 히스토리에 보존된다 (→ 고민 6).

### 배운 점
- 정확성 체크를 "누가"의 문제로 두면 답이 없다. **"무엇을 기계가, 무엇을 사람이"로 쪼개는 순간** 각각의 답은 자명해졌다.

## 고민 3. 어떤 공간에 데이터를 쌓고 보여줄 것인가 — 어떤 현행 기술을 쓸 수 있나

**슬라이드 헤드라인: "그래프 DB 없이 그래프를 — 파일이 진실, DB는 인덱스"**

### 제약 조건 (게임 회사 QA 환경)
- 게임 스펙은 사외 반출 불가 → SaaS·클라우드 LLM만 전제할 수 없음.
- QA 담당자 PC에 설치되어야 함 → 서버 운영 부담 없어야 함.
- 지식은 사람도 읽고 고칠 수 있어야 함 → 블랙박스 DB만으로는 안 됨.

### 기술 선택과 그 이유
| 계층 | 선택 | 이유 / 기각한 대안 |
|------|------|------|
| 앱 | Tauri 데스크톱 (Rust + React) | 로컬 완결, 인스톨러 배포. Electron 대비 가벼움 |
| 지식 원본 | Markdown 파일 (`db/*.md`) | **사람이 읽고 diff 볼 수 있는 것이 진실의 원천**. 에디터·git과 호환 |
| 그래프/인덱스 | SQLite (knowledge.sqlite) | Neo4j 등 그래프 DB 검토 후 **기각**: 규모가 수천~수만 assertion, traversal 깊이 3홉 이내라 재귀 CTE로 충분. 파일+DB 원자 커밋은 SQLite 트랜잭션이 보장하는데 외부 graph DB로 가면 오히려 깨짐. 데스크톱 배포에 DB 서버는 부담 |
| 벡터 검색 | LanceDB (임베디드) | 별도 벡터 DB 서버 불필요 |
| 시간축/공유 | git (CLI subprocess) | 버전 관리를 직접 구현하지 않는다. 히스토리·협업·복원 전부 git에 위임 |
| LLM | 프로바이더 불가지론 — Claude/Codex/Gemini CLI, OpenAI/Anthropic API, 로컬(vLLM/llama.cpp) | 사내 보안 요건별로 갈아끼울 수 있어야 함. 로컬 모델의 현실적 제약(Qwen3 계열 chat template이 "system 메시지는 맨 앞만" 강제 → 언어 지시를 system이 아니라 마지막 user 메시지 앞에 삽입)까지 흡수 |

### 보여주는 공간
- 6개 화면: Chat(질의응답) / Graph(그래프·엔티티·지식 관리) / Review(충돌 검수) / History(git 히스토리) / Casemap(경우의 수 맵핑) / Persona(페르소나 시나리오).
- 입력 경로도 여럿: 파일 주입, URL 추출(Readability 변환), 브라우저 확장 web clipper(로컬 HTTP 서버로 수신).

### 배운 점
- "현행 기술"의 답은 신기술 도입이 아니라 **경계 설정**이었다: 파일 = 진실, SQLite = 인덱스, git = 시간, LLM = 교환 가능한 부품.

## 고민 4. 같은 지식을 여러 차원으로 — 하나의 그래프로 모든 목적을 감당할 수 있나, 차원 간 이동은?

**슬라이드 헤드라인: "그래프는 하나가 아니라 여러 개 — 엔티티가 차원 사이의 포탈이다"**

### 문제
- 지식 그래프를 참고해 LLM이 게임을 플레이(또는 테스트 시나리오 수행)하게 하려면, 같은 지식이라도 목적마다 다른 물음에 답해야 한다. "스킬 A"는 밸런스 관점에선 수치 노드, 플레이 관점에선 행동 선택지, 케이스 설계 관점에선 조합 축이다.
- 하나의 거대 그래프에 다 넣으면 목적별 탐색이 서로를 오염시키고, 목적별로 완전히 분리하면 같은 지식이 중복 축적된다.

### 지금의 답 — 복수 그래프 + 공유 엔티티 + 제한된 차원 이동
1. **그래프 = 차원.** 프로젝트 안에 목적별 그래프를 여러 개 둔다. 이름의 목적 접두사가 차원을 선언한다: `combat_*`, `quest_*`, `casemap_*`(경우의 수), `persona_*`(페르소나 시나리오), `playlog_*`(플레이 로그). 새 차원이 필요하면 LLM이 명명 규칙 안에서 새 그래프를 제안한다 (→ 고민 1).
2. **엔티티는 전 차원 공유.** "스킬 A"라는 엔티티는 하나만 존재하고, 여러 그래프에 노드로 참여한다(`graph_nodes`가 엔티티↔그래프 다대다 연결). 같은 지식을 차원마다 다시 쌓지 않는다.
3. **차원 간 이동 = traversal의 graph switch.** 그래프 탐색은 공유 엔티티를 만나면 다른 그래프로 갈아탈 수 있되, 홉 비용 상한(`maxCost=3`)과 **그래프 전환 횟수 상한(`maxGraphSwitches=2`)**을 둔다. "전투 차원에서 출발해 퀘스트 차원까지는 넘어가되, 무관한 차원으로의 폭주는 차단"이 수치 두 개로 표현된다.
4. **화면(목적)마다 차원 스코프를 잠근다.** Casemap/Persona 탭의 챗은 같은 ChatPanel을 재사용하되 `casemap_`/`persona_` 접두사 그래프만 보도록 스코프를 제한한다. 문서 검색과 그래프 컨텍스트가 항상 같은 그래프 집합을 보도록 스코프 해석을 한 곳(`resolveAllowedGraphIds`)에 모았다.
5. **LLM 플레이를 향한 차원 — playlog.** 플레이 영상·입력 로그를 (입력 전 상태 / 입력 / 목적 / 입력 후 사건) 단위 문서로 자산화하고(→ 고민 13), 여기서 나온 `스킬A --HAS_EFFECT--> 넉백` 같은 triple을 `playlog_` 차원에 쌓는다. "상태에서 행동을 고르는" 플레이 지식은 이 차원에서 소비한다.

### 배운 점
- "하나의 그래프로 모든 목적"은 틀린 목표였다. 맞는 목표는 **"하나의 엔티티 집합 위에 여러 그래프"** — 중복 없는 축적과 목적별 탐색을 동시에 얻는다.

## 고민 5. 자연어 입력의 언어 정규화는 어느 수준까지 — 용어 사전을 어떻게 관리하나

**슬라이드 헤드라인: "사전은 선불이 아니라 후불 — 정규화 최소주의로의 후퇴"**

### 시도의 역사 (실패의 기록)
1. **W3C 등 표준 어휘 사전 도입 검토** → 게임 고유 조어(스킬명·시스템명·프로젝트 은어)를 표준 사전이 커버할 리 없다. 추상 계층으로는 쓸 수 있어도 매핑 비용이 크다.
2. **언리얼 엔진 용어 기반 정규화** → 자연어 입력을 사전 용어와 매칭하는 과정 자체가 LLM 호출이고, 사전이 커질수록 매칭 프롬프트가 컨텍스트를 소진했다. **배보다 배꼽이 커져서 실패.**
3. **자체 엔티티 사전(entity-dict.json) + fuzzy 매칭 + 사용자 확인 카드** → 만들어서 운영까지 해봤다. "이 이름이 기존 엔티티와 비슷한데 같은 것인가?"를 ingest마다 사용자에게 물었더니 확인 카드가 폭증하고, 결정 부담이 검수 병목이 됐다. mid-dev 리팩터링에서 **전면 제거.**

### 지금의 답 — 정규화 최소주의
- **자동 판정은 보수적으로**: trim + NFKC 정규화 + 소문자화한 문자열의 **완전 일치만** 같은 엔티티로 본다. fuzzy 자동 병합 없음, 사용자 확인 없음.
- **입력 시 표기 통일로 완전일치 적중률을 올린다**: 추출 단계에서 `normalizeEntityName()`이 후행 괄호 부기 제거·공백 정리 등으로 표기를 통일해 완전일치 dedup에 흡수시킨다.
- **갈라진 중복은 사후 정리**: 같은 타입 안에서 정규화 동일·부분 포함·편집거리 기반으로 병합 후보를 점수화해 추천(`suggest_entity_merges`)하고, 사람이 Graph 화면에서 수동 merge한다. 병합은 한 트랜잭션으로 모든 참조(assertion, 별칭, 페이지 연결)를 이관한다.

### 배운 점
- 용어 사전은 **먼저 구축하는 자산이 아니라 나중에 정리되는 결과물**이었다. 입력 시점에 완벽한 정규화를 강제하면 (a) LLM 컨텍스트 낭비 (b) 사용자 확인 폭증 중 하나로 반드시 무너진다. "일단 보수적으로 쌓고, 중복은 싸게 발견해서 싸게 합친다"가 유지 가능한 균형점.

## 고민 6. 그땐 맞았지만 지금은 틀린 데이터 — 누가 결정하고, 결정권자에게 얼마나 부담을 줄 것인가

**슬라이드 헤드라인: "결정은 사람이, 그러나 결정의 단위를 줄이고 '보류'를 허하라"**

### 문제
- 낡은 지식의 폐기·갱신을 LLM이 자동으로 하게 두면 오판이 축적되고, 사람이 하게 하면 검수 부담이 도입 장벽이 된다. 결정권자(QA 담당자)의 부담 총량을 어떻게 설계하나.

### 지금의 답 — 결정 비용 최소화 장치들
1. **결정 단위를 섹션으로 축소.** 문서 전체가 아니라 충돌 난 섹션 하나의 existing/incoming 2컬럼 diff만 보여준다. 같은 페이지의 비충돌 섹션(신규 포함)은 자동 병합되어 즉시 반영 — 사람은 진짜 갈등 지점만 본다.
2. **같은 제안의 반복은 병합.** 벌크 ingest에서 동일 제안이 반복 발생하면 카드를 dedupe해 폭증을 막는다.
3. **즉답을 강요하지 않는다.** Reject가 곧 삭제가 아니다 — Reject를 누르면 Discard(폐기, 사유를 rejection-log에 기록) / **Pending(보류)** 2단계로 갈라진다. 보류함의 항목은 나중에 재검토(Re-review)하거나 승격(Promote)하거나 폐기할 수 있다. "지금은 판단 못 하겠다"가 합법적인 선택지다.
4. **폐기가 아니라 대체.** 카디널리티 충돌을 승인하면 기존 assertion은 삭제되지 않고 `superseded`로 강등된다. "그땐 맞았던" 기록이 그래프에 남는다.
5. **시간축은 git이 진다.** DB는 현재 스냅샷만 유지하고, 과거 상태는 전부 git 커밋에 있다. `version_comparison` 질문 타입은 사용자가 과거 커밋 시점을 골라 현재와 비교하게 한다 — "그땐 뭐라고 되어 있었지?"를 시스템에 물을 수 있다.
6. **커밋 시점도 사용자가 결정.** 자동 커밋을 제거하고 "Sync to Remote"를 누른 시점에만 커밋·push한다. 언제가 "공유할 만한 상태"인지도 결정권자의 몫으로 남겼다.

### 배운 점
- "누가 결정할 것인가"의 답은 처음부터 사람이었다. 진짜 설계 대상은 **결정의 단위·빈도·되돌림 가능성**이었다. 부담은 결정 개수가 아니라 "잘못 결정했을 때의 공포"에서 오고, 보류와 히스토리 복원이 그 공포를 줄인다.

---

## 고민 7. 들어오는 데이터 타입에 맞게 축적하자 — 같은 종류의 데이터는 같은 포맷으로

**슬라이드 헤드라인: "포맷 통일은 프롬프트가 아니라 파서가 강제한다"**

### 문제
- 패치노트·스킬 스펙·몬스터 스펙이 제각각의 자유 문서로 쌓이면, 나중에 "쿨타임 있는 스킬 전부"를 물어도 문서마다 구조가 달라 재활용이 안 된다.

### 지금의 답 — 2계층 포맷 강제
1. **데이터 타입 스키마 (내용 구조)**: `data_types/*.yaml`로 13종을 선언 — patch_note, skill_spec, monster_spec, item_spec, quest_spec, zone_spec, screen_spec, interaction_spec, state_spec, player_spec, session_spec, casemap, persona. 각 타입은 `fields`(필드명: 설명) 목록이고, ingest 시 타입을 선택하면 LLM의 자유 분해 대신 **필드 채우기 구조화 추출**이 실행된다.
   - 못 찾은 필드는 "정보 없음"으로 표기 — 지어내지 않는다.
   - 전 필드가 비면 문서 자체를 만들지 않고 스킵 — 빈 껍데기 문서 방지.
   - 한 파일에 타입 N개를 지정하면 타입마다 독립 문서가 생성된다 — 같은 원본에서 관점별 문서.
   - 타입 정의는 git에 커밋되어 프로젝트(팀) 공유 자산이 된다.
2. **Markdown v2 (문서 구조)**: 어떤 경로로 들어온 문서든 파서가 동일 구조를 강제한다 — frontmatter의 `page_id`/`title`/`page_type`/`summary`, 정확히 하나의 H1, 모든 H2에 안정 섹션 ID(`{#sec-ULID}`), frontmatter와 본문의 양방향 검증. 검증을 통과하지 못하면 저장 자체가 안 된다.

### 배운 점
- "포맷을 맞추자"고 프롬프트에 아무리 써도 LLM 출력은 흔들린다. 재활용 가능한 축적은 **스키마 선언(YAML) + 기계 검증(파서)** 두 층이 받쳐야 나왔다.

## 고민 8. 답변에 맞게 데이터를 검색하자 — 답변이 요구하는 요소를 나열하고 요소별로 찾게 하자

**슬라이드 헤드라인: "답변을 요소로 분해하면, 검색과 역질문이 저절로 정의된다"**

### 문제
- "관련 문서 검색 → 통짜 프롬프트 → 자유 서술 답변"은 답이 그럴듯할 뿐, 어떤 요소가 근거 있고 어떤 요소가 창작인지 구분이 안 된다.

### 지금의 답 — 요소 분해 계약
1. **답변 요소를 YAML로 선언**: 질문 타입마다 `fields`(예: change_impact의 `affected_pages`, `update_candidates`)를 선언하고, 답변은 이 요소들을 채운 **JSON으로 통일**했다(자유 md 답변 폐지). LLM은 요소 하나하나에 대해 근거를 찾아 채운다.
2. **근거는 citation으로 강제**: 검색된 섹션마다 랜덤 키(`[CIT:xxxxx]`)를 발급하고, LLM은 발급된 키만 인용할 수 있다. 답변 완료 후 마커를 스캔해 발급된 키만 구조화 인용으로 확정 — 지어낸 인용은 걸러진다. 근거 섹션이 없으면 "없다"고 답하게 하는 규칙이 프롬프트에 조건부로 들어간다.
3. **검색 자체의 진화 (3단계)**:
   - 토큰 매칭의 한계에서 출발 — 한국어 조사가 붙은 토큰("결투장을")은 본문의 "결투장"과 exact 매치에 실패하고, 불용어("어떤", "시스템")도 후보를 만든다. 너무 느슨한 동시에 너무 깨지기 쉬웠다.
   - 그래프 확장 — 문자열로 못 찾은 문서를 그래프 traversal이 승격한다. lexical 후보에서 출발해 관계를 따라간 지점의 evidence 섹션을 후보에 추가. 이때 predicate 축 필터(예: 의존/영향 관계만 따라가기)로 목적에 맞는 확장만 한다.
   - embedding 시맨틱 검색 병합 — lexical의 랭킹 공백을 임베딩 검색으로 메꾼다(LanceDB 인덱싱은 이미 있었으므로 소비 경로를 연결).
4. **없는 요소는 역질문**: 질문 타입에 `required_info`(입력 계약)를 선언 — 예: change_impact는 "무엇을 / 어떻게 바꾸는가 / 의존 관계"가 필요하다. 위키에서 못 찾은 항목은 LLM이 사용자에게 **인라인 입력 폼**(라디오/텍스트/파일/링크 + "없음/모름" 필수 옵션)으로 되묻고, 받은 답은 이번 답변의 컨텍스트로 주입될 뿐 아니라 **위키/그래프에 축적**된다 — 질의응답이 곧 지식 수집 루프가 된다.

### 배운 점
- "검색을 잘하자"보다 "**답변이 무엇으로 구성되는지를 먼저 계약하자**"가 순서였다. 요소가 선언되니 (a) 무엇을 검색할지 (b) 무엇이 부족한지 (c) 무엇을 되물을지가 전부 그 계약에서 파생됐다.

## 고민 9. 데이터 축적의 목적을 개별 답변 타입에 맞추자 — 답변마다 답변 파이프라인을 구축하자

**슬라이드 헤드라인: "13개의 희망사항을 4개의 계약으로 — 답할 수 없는 타입은 기능이 아니라 거짓말"**

### 문제
- 초기에 QA 업무에서 "있으면 좋겠는 답변" 목록으로 질문 타입을 13개까지 만들었다: balance_simulation, regression_test, test_gap, policy_violation, design_flaw_detection, opinion_delivery, risk_based_recommendation, condition_based_test, combination_explosion_test, change_impact...
- 그런데 모든 타입이 동일한 검색을 쓰고, 타입은 system prompt에 "출력 형식" 한 조각만 덧붙이고 있었다. 형식만 다르고 재료는 같은 답변들.

### 전환점 — "이 타입은 답할 수 있는 데이터가 쌓이고 있는가" 진단
각 타입이 답하려면 필요한 데이터를 실제 축적 파이프라인과 대조했더니 구조적 공백 4개가 나왔다:
- **공백 A**: relation에 의존/영향 의미축이 없다 — `ATTACKS` 같은 도메인 서술 predicate만 있고 "변경이 전이되는가(`DEPENDS_ON`/`AFFECTS`)"가 없어 영향 분석이 불가능.
- **공백 B**: 시간축이 없다 — DB는 현재 스냅샷만 유지(upsert)해서 과거 상태는 git에만 있는데 검색이 git을 안 본다. 버전 비교가 구조적으로 불가능.
- **공백 C**: 그래프가 후보 확장에 안 쓰인다 — traversal이 이미 찾은 후보의 라벨 계산에만 쓰이고 새 후보를 추가하지 않았다.
- **공백 D**: 그래프 컨텍스트가 프롬프트에 아예 안 실린다 — LLM이 assertion을 한 줄도 보지 못하고 있었다.

### 결단
1. **답할 수 없는 타입 9개 삭제.** 데이터가 받쳐주지 않는 타입은 존재 자체가 사용자 기만이다.
2. **답하게 만들 수 있는 4개로 통폐합**: `change_impact`(변경 영향), `new_system_impact`(신규 시스템 영향), `related_content`(연관 콘텐츠), `version_comparison`(버전 비교).
3. **타입 = 파이프라인 선언.** 각 타입 YAML이 출력 형식(fields)만이 아니라 **retrieval 전략까지 선언**한다: `graph_expand`(홉 수), `predicate_axes`(따라갈 관계 축), seed 전략, `include_history`(git 히스토리 주입), `scope`(범위 선택 UI 노출). UI 분기는 타입 id 하드코딩 없이 YAML 플래그로만.
4. **축적도 소비에 맞춰 교정.** ingest 프롬프트에 의존/영향 축 predicate 추출 지시를 추가 — 쌓기만 하고 안 쓰는 데이터, 쓰려는데 안 쌓이는 데이터를 없앤다. `zero_residue_meaning`(잔존 0의 의미 — 예: change_impact에서 "영향 격리 완료") 같은 타입별 완료 조건도 함께 선언.

### 배운 점
- 답변 타입은 UI 기능이 아니라 **데이터 계약**이다. "이 답변을 하려면 무엇이 쌓여 있어야 하는가"를 먼저 묻지 않은 타입은 전부 삭제 대상이 됐다. 기능을 줄이는 결정이 이 프로젝트에서 가장 생산적인 결정이었다.

## 고민 10. (추가) LLM이 지어낸 지식이 위키를 오염시키면? — 환각의 차단선

**슬라이드 헤드라인: "모든 지식에 원문 인용을 물려라 — quote가 없으면 지식이 아니다"**

### 문제
- 이 시스템은 LLM이 지식을 **쓰는** 쪽에도 관여한다(분해·추출·그래프 할당). LLM의 환각이 답변에서 나오면 한 번 틀리고 말지만, 지식베이스에 들어가면 **영구 오염**이고 이후 모든 답변의 재료가 된다.

### 지금의 답 — 들어올 때와 나갈 때 이중 차단
- **들어올 때 (ingest)**:
  - 모든 assertion은 근거 quote가 원문 섹션의 **exact substring**이어야 한다. 아니면 전체 거부.
  - 구조화 추출에서 못 찾은 필드는 "정보 없음" — 채우려고 지어내지 않는다. 전부 비면 문서 생성 자체를 스킵.
  - 문서 본문은 LLM의 요약·재서술이 아니라 **원문 verbatim**을 보존한다. LLM이 결정하는 건 "어디를 자를지, 어디에 둘지"이지 "무슨 내용인지"가 아니다.
  - 배치 경로(page_path)가 불확실하면 추측으로 채우지 않고 그 chunk를 실패 처리한다 — 폴백으로 조용히 넘어가는 경로를 없앴다.
- **나갈 때 (답변)**:
  - 발급된 citation 키만 인용 가능, 사후 스캔으로 미발급 키 무효화 (→ 고민 8).
  - 관련 문서/그래프 근거가 없으면 "찾지 못했다"고 답하라는 규칙이 조건부 주입된다.
  - 인사말 같은 비질문 입력은 규칙 기반으로 감지해 검색·인용 파이프라인 자체를 건너뛴다 — 억지로 검색해서 억지로 인용하는 환각 기회를 차단.

### 배운 점
- 환각 대책의 핵심은 더 좋은 프롬프트가 아니라 **검증 가능한 형태로만 지식을 받는 것**이었다. "출처를 대라, 원문과 대조하겠다"가 기계적으로 가능해지는 순간 환각은 대부분 형식 검증에서 걸러진다.

## 고민 11. (추가) 같은 지식이 다시 들어왔을 때 "같은 것"임을 어떻게 아나 — 문서 정체성 문제

**슬라이드 헤드라인: "제목은 정체성이 아니다 — 안정 ID로의 이행"**

### 문제
- 위키는 같은 주제의 문서가 반복적으로 갱신되는 시스템이다. 새로 들어온 섹션이 기존 섹션의 "갱신"인지 "다른 지식"인지 판정해야 충돌 검출(고민 2·6)이 성립한다.

### 시도와 실패
- 초기: **heading 텍스트 비교**로 같은 섹션을 판정했다. LLM이 제목 표현을 조금만 바꿔도("고블린 전사 - 약점" vs "고블린 전사의 약점") 다른 섹션으로 취급되어 중복이 누적되고, 반대로 우연히 같은 제목이면 무관한 내용이 충돌로 잡혔다.

### 지금의 답
- **정체성과 표시명의 분리.** 페이지는 `page_id`, 섹션은 `sec-ULID`라는 안정 ID를 갖고, 충돌 판정·병합·evidence 연결 전부 ID 기준이다. ID는 LLM이 아니라 파서·직렬화 계층이 생성·유지한다.
- 표시명(title/heading)은 최초 ingest 때 한 번만 생성하고 이후 수정에서 재생성하지 않는다 — 문서를 고쳤다고 이름이 흔들리지 않는다.
- 표시명 자체도 명명 규칙(제목은 `주어-출현-역할-속성` 최대 4요소 약 40자, 섹션은 `대상-측면`)을 프롬프트로 강제 — 사람이 목록을 훑을 때 의미가 읽히는 이름.

### 배운 점
- "무엇이 같은 문서인가"는 사소해 보이지만 충돌 검출·근거 연결·수정 승인 전부가 이 판정 위에 서 있었다. **변하는 것(텍스트)에 정체성을 두면 시스템 전체가 흔들린다.**

## 고민 12. (추가) 문서 따로, 그래프 따로 — 두 저장소가 어긋나면 누구 말을 믿나

**슬라이드 헤드라인: "두 개의 진실은 하나의 트랜잭션으로"**

### 문제
- 지식이 문서(파일)와 그래프(DB) 두 곳에 쓰이므로(고민 1), 한쪽만 성공하는 순간이 반드시 생긴다 — "문서는 저장됐는데 그래프엔 없다", 그 역도. 초기 구현은 실제로 "파일 먼저 쓰고 그래프는 나중에 sync"하는 2단계여서 크래시·실패 시 드리프트가 발생했다.

### 지금의 답
- **단일 원자 커밋**: 한 chunk의 페이지 저장(파일시스템)과 assertion 생성(SQLite)을 journal 기반 하나의 트랜잭션으로 처리한다. 임시 파일에 먼저 쓰고, DB 트랜잭션 커밋 후 실제 경로에 적용, 중간 실패 시 둘 다 반영 안 됨. 비정상 종료는 프로젝트 오픈 시 journal을 읽어 복구.
- **수정 승인도 섹션 단위 재동기화**: 사람이 충돌 섹션을 승인하면 그 섹션에서만 관계를 재추출하고, 그 섹션을 근거로 삼던 기존 assertion만 폐기·교체한다. 다른 섹션의 근거로도 지지되는 assertion은 보존 — 문서 수정이 그래프에 정확히 그만큼만 전파된다.

### 배운 점
- 이중 표현(고민 1)을 선택한 순간 이 문제는 예약되어 있었다. **표현을 두 개 두기로 했다면 커밋은 하나여야 한다.** 그리고 이 요구가 기술 선택(고민 3)을 역으로 규정했다 — 외부 graph DB를 쓰면 이 원자성이 깨진다.

## 고민 13. (추가) 텍스트가 아닌 지식은 어떻게 하나 — 플레이 영상·스크린샷·입력 로그

**슬라이드 헤드라인: "이미지는 저장하지 않는다, 서술한다"**

### 문제
- QA의 가장 풍부한 지식원은 문서가 아니라 **플레이 그 자체**다 — 녹화 영상, 키 입력 로그, 스크린샷. 그런데 파이프라인 전체(검색·추출·인용)가 "본문은 텍스트"라는 전제 위에 있다. 이미지를 md에 경로나 base64로 넣으면 검색도 추출도 안 되는 죽은 자산이 된다.

### 검토한 선택지
| 방식 | 결과 |
|------|------|
| 이미지 경로/base64 첨부 | 기각 — 검색·추출 불가능한 죽은 자산 |
| SVG 코드로 변환 | 기각 — 스크린샷은 사진적 정보라 벡터 변환이 애초 불가능, 나와도 의미 있는 텍스트가 아님 |
| **서술형 텍스트 변환** | **채택** — 멀티모달 LLM이 전/후 스크린샷 쌍을 판독해 "체력 40%, 몬스터 3마리 근접" 같은 자연어로 서술. 기존 검색·추출 파이프라인을 그대로 통과 |
| ASCII 아트 보조 | 채택 — 정보 밀도는 낮지만 화면 배치 이해에 보조로 유용 |

### 지금의 답 — Stage 0 (파이프라인 앞단의 변환 계층)
- (입력 전 스크린샷, 입력 후 스크린샷, 키 입력, 입력 목적, 발생 사건) 레코드를 멀티모달 LLM 1회 호출로 "입력 전 상태 / 입력 / 목적 / 입력 후 사건" 문단의 일반 문서로 변환 — 이후는 기존 파이프라인이 그대로 처리해 `스킬A --HAS_EFFECT--> 넉백` 같은 triple이 자연스럽게 나온다.
- 사람이 이미 기록한 값(입력 목적)은 LLM이 재해석하지 않고 그대로 인용한다.
- 원본 이미지의 경로도 데이터도 본문에 남기지 않는 것이 원칙 — 텍스트 전제를 지킨다.
- 이 차원의 지식은 `playlog_` 그래프에 쌓인다 (→ 고민 4의 "LLM이 플레이하게 하기"의 재료).

### 배운 점
- 멀티모달 지식의 정답이 "멀티모달 저장소"가 아닐 수 있다. **파이프라인의 전제(텍스트)를 지키는 최소 변환 계층** 하나가, 저장소 전체를 멀티모달로 개조하는 것보다 압도적으로 쌌다.

## 고민 14. (추가) 혼자 쓰는 도구인가, 팀의 지식베이스인가 — 공유와 이식의 문제

**슬라이드 헤드라인: "동기화는 git에, 교환은 zip에 — 지식의 소유권은 프로젝트 폴더에"**

### 문제
- 개인 PC의 로컬 앱(고민 3)으로 시작했지만, QA 지식은 본질적으로 팀 자산이다. 여러 담당자가 쌓은 지식을 어떻게 합치고, 다른 도구·프로젝트에서 쌓은 지식을 어떻게 들여오나.

### 지금의 답
1. **팀 동기화 = git remote 공유.** 프로젝트 폴더 자체가 git 저장소다. 자동 커밋 없이 사용자가 "Sync to Remote"를 누르면 그 시점 상태를 커밋 → `pull --rebase`로 남의 변경 통합 → push. rebase 충돌은 앱 안에서 파일 목록+diff로 보여주고 해결/중단을 선택하게 한다. 그래프도 스냅샷(graph.json)으로 export되어 같은 커밋에 실린다.
2. **도구 간 교환 = `.llmwiki` 패키지.** 문서(`db/**/*.md`) + 그래프 스냅샷(graphs.json)을 담은 표준 zip 양식을 공개 스펙(interchange-spec)으로 정의 — 외부 앱·스크립트로 쌓은 지식도 이 양식만 맞추면 import된다. 스냅샷의 노드 ID는 import 시 재발급하고, 중복은 stable key로 병합.
3. **질문 타입·데이터 타입도 공유 자산.** `question_types/`, `data_types/` YAML이 git에 커밋되어 팀이 같은 답변 계약·데이터 계약을 쓴다. 개인 오버라이드는 별도 디렉터리로 분리(tombstone으로 팀 타입 숨기기도 가능).

### 배운 점
- 협업 기능을 자체 서버로 만들지 않고 git에 위임한 것은 옳았다 — 버전·병합·권한·백업을 공짜로 얻는다. 대신 **"커밋 단위 = Sync 단위"라는 굵은 granularity를 수용**해야 했고, 이는 버전 비교(고민 6·9)의 비교 시점이 "사용자가 고르는 Sync 시점"이 된 이유이기도 하다.

## 고민 15. (추가) 소거법의 기록 — 실패 로그·잘못된 루트를 LLM에게 어떻게 전달하고, 컨텍스트를 어떻게 아끼고, 소거는 언제까지 유효한가

**슬라이드 헤드라인: "부정 지식도 자산이다 — 단, 원문이 아니라 결정으로 압축하고, 유효기간은 짧게 잡아라"**

### 문제
- 복잡한 QA 업무(버그 원인 좁히기, 경우의 수 배제)는 본질적으로 소거법이다. 정답 하나에 도달하기까지 "아닌 것"의 기록이 훨씬 많이 쌓인다.
- 그런데 이 실패 로그·기각 기록을 어떻게 다루나. 세 가지가 한꺼번에 걸린다:
  1. **전달** — 안 주면 LLM이 이미 소거된 루트를 반복 제안한다 (같은 modification 카드가 재-ingest 때마다 되살아나는 실제 경험).
  2. **컨텍스트 예산** — 실패는 성공보다 많다. 실패 로그 원문을 다 주입하면 컨텍스트가 부정 지식으로 먼저 소진된다 (고민 5에서 용어 사전 매칭이 컨텍스트를 소진해 실패한 것과 같은 함정).
  3. **유효기간** — 한 번 소거된 루트는 영원히 소거된 것인가? 게임은 패치된다. 그때 아니었던 루트가 지금은 정답일 수 있다. 긍정 지식의 만료(고민 2·6)와 거울상인 문제.

### 지금의 답 — 소거의 기록·재주입·압축
1. **폐기는 삭제가 아니라 기록이다.** 사용자가 제안을 Discard하면 그냥 사라지는 게 아니라 `rejection-log.jsonl`에 한 줄(시각, 대상 경로, 출처 파일·범위, **사유**)이 append된다. "왜 아닌지"가 소거의 자산으로 남는다.
2. **같은 길목에서만 되살린다 (dismissal context).** 같은 소스를 다시 ingest할 때 이 로그를 프롬프트에 주입하되, 지시는 명확하다 — "아래의 과거 기각 결정과 겹치는 구간은 페이지를 제안하지 말고 'Skipped (prior dismissal)'로 보고하라". 소거된 루트를 LLM이 다시 밟지 않게 하는 동시에, 무엇을 건너뛰었는지는 사람에게 투명하게 보고된다.
3. **컨텍스트는 3중으로 아낀다.**
   - **관련성 스코프**: 전체 로그가 아니라 지금 처리 중인 소스 파일과 일치하는 항목만 주입.
   - **최근성 컷**: 로드는 최신 50건, 주입은 20건 상한 — 로그가 아무리 쌓여도 프롬프트 비용은 상수.
   - **압축 표현**: 실패의 원문(기각된 draft 전체)이 아니라 결정 요약 한 줄(`대상 경로 ← 출처:범위 — 사유`)만 주입. 부정 지식은 "무엇이 어디서 왜 기각됐나"면 충분하고, 내용 전문은 필요 없다.
   - 원칙 하나 추가: **보류(Pending) 항목은 의도적으로 주입하지 않는다** — 사람이 아직 판단하지 않은 것이 모델의 다음 판단에 영향을 줘선 안 된다. 소거로 취급되는 것은 사람이 명시적으로 기각한 것뿐이다.
4. **다른 층위의 소거 장치들도 같은 원리를 따른다.**
   - Casemap 위저드: 경우의 수 조합에서 배제(excluded)된 케이스를 걸러내고, 확정 테스트 플랜에는 비배제 케이스만 기록한다 — 조합 폭발을 소거로 다스리는 화면.
   - 역질문 루프의 "없음/모름" 마킹(고민 8): "이 정보는 사용자도 없다더라"라는 부정 정보인데, 유효기간을 **대화 단위로만** 의도적으로 제한했다 — 다음 대화에서는 상황이 바뀌었을 수 있으니 다시 물을 수 있어야 한다.
   - `zero_residue`(잔존 0): 질문 타입마다 "더 소거할 것이 없음"의 의미를 선언한다 — 예: change_impact에서 잔존 0 = 영향 격리 완료. 소거의 **종료 조건**을 타입별 계약으로 명시.

### 한 번 소거된 루트는 언제까지 유효한가 — 현재의 답과 남은 질문
- 지금 시스템의 유효기간은 세 층위다:
  | 부정 지식 | 유효기간 | 만료 방식 |
  |------|------|------|
  | "없음/모름" 마킹 | 가장 짧음 — **대화 단위** | 대화가 끝나면 자동 소멸, 다음 대화에서 재질문 가능 |
  | rejection-log 항목 | 중간 — **최근성** | 최신 50건 창에서 밀려나면 자연 만료 (명시적 TTL 없음) |
  | Pending 보류 항목 | 명시 해제까지 | 사람이 재검토(Re-review)·승격(Promote)으로 직접 되살림 |
- 원리적으로 소거는 무조건적 사실이 아니라 **조건부 사실**이다 — "그 버전, 그 조건에서 아니었다". 패치가 나오면 닫힌 루트는 다시 열릴 수 있다. 긍정 지식의 만료가 모순 주도였듯(고민 2), 부정 지식의 만료도 결국 **재검증 주도**가 되어야 한다: 소거 기록에 조건(당시 버전/커밋)을 물려 두고, 그 조건이 변한 것이 감지되면 소거를 "풀 후보"로 승격하는 것. 시간축이 이미 git에 있으므로(고민 6) 재료는 있고, 현재의 최근성 컷은 그 근사치다 — 남은 숙제.

### 배운 점
- 실패 로그의 가치는 원문이 아니라 **결정과 사유**에 있다. 압축하지 않은 부정 지식은 컨텍스트를 잡아먹는 짐이고, 압축된 부정 지식은 같은 실수를 막는 가드레일이다.
- 그리고 부정 지식은 긍정 지식보다 **유효기간을 짧게 전제하는 쪽이 안전**하다 — 소거를 너무 오래 믿으면 열린 루트를 영영 안 가보게 되고, 재검증 비용은 상대적으로 싸다.

## 맺음말 슬라이드 후보

- 관통하는 교훈 셋:
  1. **LLM에게 결정을 맡기지 말고 제안을 맡겨라** — 배치·추출·병합 후보까지는 LLM, 확정은 검증기와 사람.
  2. **축적과 소비를 같은 계약으로 묶어라** — 답할 수 없는 데이터를 쌓는 것도, 쌓이지 않은 데이터로 답하는 것도 전부 여기서 무너졌다.
  3. **기능을 지우는 결정이 가장 생산적이었다** — 질문 타입 13→4, fuzzy 사전 제거, 이미지 저장 포기.
