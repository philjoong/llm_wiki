# LLM Wiki

<img src="logo.jpg" alt="LLM Wiki" width="120" align="right" />

**LLM과 함께 개인 지식 베이스를 구축하고 관리하는 데스크톱 앱** (Tauri 기반, Windows/macOS/Linux)

파일·웹페이지 등 원본 자료를 넣으면 LLM이 내용을 분해해 위키 문서(`db/*.md`)와 지식 그래프를 자동으로 구축합니다. 이후 하이브리드 검색(토큰 + 벡터) + 그래프 컨텍스트 기반으로 위키에 질문하고 답변을 받을 수 있습니다. QA 업무를 위한 테스트 플랜 생성(경우의 수 맵핑), 페르소나 시나리오 생성 기능도 포함합니다.

## 설치 및 실행

### 배포판 사용

릴리스 페이지에서 설치 파일을 받아 실행합니다. 앱 내 **설정 > 정보** 탭에서 업데이트 확인 저장소(GitHub `owner/repo` 또는 GitLab URL)를 지정하면 새 버전 알림을 받을 수 있습니다.

### 소스에서 실행

요구 사항: Node.js, Rust 툴체인 (Tauri 2)

```bash
npm install
npm run tauri dev     # 개발 모드 실행
npm run tauri build   # 배포용 빌드
```

## 시작하기 — 프로젝트 만들기

앱을 실행하면 **Select Project** 화면이 나타납니다. 프로젝트는 Git 브랜치 단위로 관리됩니다.

- **원격 Git URL**: 상단에 원격 저장소 URL(예: `gitlab.example.com/group/repo.git`)을 입력하고 새로고침하면 원격 브랜치 목록이 표시됩니다. 브랜치를 선택하면 연결할 로컬 폴더를 지정한 뒤 프로젝트가 열립니다.
- **새 프로젝트 만들기**: 브랜치 이름을 입력하고 **Create**를 누르면 새 브랜치/프로젝트가 생성됩니다. 원격 URL 없이 만들면 **로컬 온리** 프로젝트가 되며(HardDrive 아이콘 + "local only" 태그), 나중에 **Push to remote** 버튼으로 원격에 올릴 수 있습니다.
- **Export as .llmwiki / Import .llmwiki**: 프로젝트 전체를 `.llmwiki` 파일 하나로 내보내거나, 받은 파일을 새 프로젝트로 가져올 수 있습니다.

프로젝트는 생성 시 자동으로 git 저장소로 초기화되며, 기본 질문 유형/데이터 유형 템플릿(`question_types/`, `data_types/`)이 함께 복사됩니다.

> ⚠️ 프로젝트를 사용하려면 먼저 **설정에서 LLM 제공자를 구성**해야 합니다.

## 설정

좌측 사이드바 하단 톱니바퀴 아이콘으로 진입합니다.

### LLM 모델

지식 추출과 답변 생성에 사용할 LLM을 설정합니다. 여러 제공자를 각각 설정해 두고 하나만 활성화하는 방식이며, 각 제공자의 API 키는 독립적으로 보관되어 전환해도 사라지지 않습니다. **연결 테스트** 버튼으로 즉시 확인할 수 있습니다.

| 제공자 | 필요 설정 |
|---|---|
| OpenAI | API 키, 모델 |
| Anthropic | API 키, 모델 |
| Google (Gemini) | API 키, 모델 |
| Ollama | Ollama URL (기본 `http://localhost:11434`), 모델 — API 키 불필요 |
| Claude Code CLI / Codex CLI / Gemini CLI | 로컬에 설치된 CLI를 subprocess로 사용 — API 키 불필요 |
| 커스텀 엔드포인트 | OpenAI 호환(`/chat/completions`) 또는 Anthropic 호환(`/v1/messages`) 엔드포인트 URL — LiteLLM, vLLM, LM Studio, LocalAI, Azure OpenAI 등 |

### 벡터 임베딩

위키에 대한 **의미 검색(RAG)** 을 활성화합니다.

- **내장 (BGE-small)**: 별도 서버 없이 번들된 fastembed 모델 사용. 첫 사용 시 ~130MB 다운로드.
- **외부 서버**: OpenAI 호환 `/v1/embeddings` 엔드포인트 직접 입력 (LM Studio, Ollama embedding 등 로컬 모델은 API 키 불필요).
- 청크 크기/오버랩 조정, 전체 페이지 재인덱싱 기능 제공.
- 비활성화 시 검색은 토큰 매칭만 사용합니다.

### 출력 환경설정

- **AI 출력 언어**: chat 답변·위키 페이지 등 AI 생성물의 언어를 강제합니다. "Auto"면 입력/원본 언어를 따릅니다.
- **대화 히스토리 길이**: 요청마다 LLM에 전달할 이전 메시지 수.

### 인터페이스

UI 언어(한국어/English)와 외관을 설정합니다.

### 질문 유형 / 데이터 유형

프로젝트의 질문 유형(`question_types/`)과 데이터 유형(`data_types/`) YAML 카탈로그를 앱 안에서 관리합니다. 형식은 [커스터마이징](#질문-유형--데이터-유형-커스터마이징) 참고.

### 정보

앱 버전, Clip 서버 상태 확인, 업데이트 확인 저장소 설정(GitHub/GitLab, private 저장소는 액세스 토큰 필요), 시작 시 자동 업데이트 확인 옵션.

## 지식 추출 (Ingest)

**"자료를 넣으면 AI가 읽고 위키로 정리해 주는" 기능**입니다. 기획서, 패치노트, 참고 웹페이지 같은 원본 자료를 넣으면, AI가 내용을 주제별로 나눠 위키 문서를 만들고 개념 간의 관계를 지식 그래프에 기록합니다. 사용자가 직접 문서를 작성하거나 분류할 필요가 없습니다.

### 자료를 넣는 3가지 방법

| 방법 | 하는 법 |
|---|---|
| **파일** | 좌측 사이드바 하단 **지식 추출** 버튼 → 파일 선택 (여러 개 선택 가능) |
| **웹페이지 URL** | **URL에서 지식 추출** 버튼 → 주소 입력. 광고·메뉴를 제외한 본문만 가져옵니다 |
| **브라우저에서 바로** | Chrome 확장 [LLM Wiki Clipper](#브라우저-확장--llm-wiki-clipper)로 보고 있는 페이지를 클립하면 앱이 자동으로 받아 처리합니다 |

### "데이터 유형을 선택하세요" 창이 뜨면?

추출을 시작하면 데이터 유형(예: 패치노트) 체크박스 목록이 표시됩니다. **자료를 정해진 양식으로 정리할지** 묻는 것입니다.

- **아무것도 선택하지 않음** (일반적인 경우): AI가 자료를 읽고 알아서 주제별 위키 페이지로 나눕니다.
- **유형 선택**: 예를 들어 "패치노트"를 선택하면 버전·적용 날짜·변경 내용 같은 미리 정의된 항목에 맞춰 한 장의 정리 문서를 만듭니다. 항상 같은 양식으로 쌓고 싶은 자료(패치노트, 버그 리포트 등)에 유용합니다. 여러 유형을 선택하면 유형별로 문서가 하나씩 만들어집니다.

양식은 [데이터 유형 커스터마이징](#질문-유형--데이터-유형-커스터마이징)에서 직접 추가할 수 있습니다.

### 추출이 시작되면 무슨 일이 일어나나요?

우측 하단 **활동 패널**에 진행 상황이 3단계(Stage 1/3 → 3/3)로 표시됩니다. 내부적으로는:

1. AI가 자료를 읽고 주제 단위로 나눈 뒤, 각각을 어느 위키 페이지에 넣을지 결정합니다.
2. "파이어볼 → 소모한다 → 마나 30" 같은 **개념 간 관계**를 뽑아 지식 그래프에 기록합니다.
3. 위키 문서(`db/` 폴더의 markdown 파일)로 저장합니다.

완료되면 **그래프 탭 > Files**에서 생성된 문서를, **Knowledge**에서 그래프를 확인할 수 있습니다.

### 알아두면 좋은 점

- **기존 문서와 내용이 부딪히면 덮어쓰지 않습니다.** 확인이 필요한 항목은 [수정 요청](#수정-요청-review-탭) 탭에 쌓이고 사이드바에 빨간 숫자 배지가 표시됩니다. 추출 후에는 이 탭을 한 번 확인하세요.
- 큰 파일은 자동으로 나눠 처리되며, 일부가 실패해도 나머지는 계속 진행됩니다.
- 작업 목록은 저장되므로 처리 중 앱을 껐다 켜도 이어서 진행됩니다.

## Chat — 위키에 질문하기

지금까지 쌓은 위키를 근거로 AI에게 질문하는 화면입니다. 인터넷 지식이 아니라 **내 프로젝트의 위키 문서를 찾아 읽고 답변**하기 때문에, 자료를 많이 넣을수록 답변이 정확해집니다.

예시: `파이어볼 스킬의 마나 소모량과 쿨타임 알려줘` → AI가 관련 위키 페이지를 검색해 읽은 뒤 답변하고, 어떤 문서를 참고했는지 출처를 함께 보여줍니다.

### 입력창 옆 버튼 2가지

- **⚡ RAG 버튼**: 켜면 단어가 정확히 일치하지 않아도 **의미가 비슷한 문서**까지 찾아냅니다. 예를 들어 "화염구"로 질문해도 "파이어볼" 문서를 찾을 수 있습니다. (설정 > 벡터 임베딩을 먼저 켜야 사용 가능하며, 꺼져 있으면 단어 일치 검색만 사용합니다.)
- **질문 유형 드롭다운**: 답변을 정해진 양식으로 받고 싶을 때 선택합니다. 예를 들어 "버그 분석" 유형을 선택하면 답변이 항상 원인/해결 방법 항목으로 정리되어 나옵니다. "None"이면 자유 형식입니다.

### 답변 화면 보는 법

- 답변은 실시간으로 출력됩니다. 길어지면 **Stop**으로 중단할 수 있고, 마음에 안 들면 **Regenerate**로 다시 생성할 수 있습니다.
- AI의 중간 사고 과정은 접힌 **"Thinking"** 섹션에 숨겨져 있으니 필요할 때만 펼쳐 보면 됩니다.
- 답변 아래 **References 패널**이 이 답변의 근거가 된 위키 페이지 목록입니다. 클릭하면 해당 문서가 바로 열리므로 **답변이 맞는지 원문으로 검증**할 수 있습니다.
- 대화 중 나온 유용한 내용은 **Wiki에 저장** 버튼으로 위키에 문서로 남길 수 있습니다.

### 대화 관리

좌측 대화 목록에서 **New Chat**으로 새 대화를 시작합니다. 대화는 프로젝트별로 자동 저장되므로 앱을 껐다 켜도 이전 대화를 이어볼 수 있습니다.

## 그래프 탭

위키에 쌓인 지식을 눈으로 확인하고 정리하는 화면입니다.

**지식 그래프란?** 자료에서 뽑아낸 개념들을 점(노드)으로, 개념 사이의 관계를 선(엣지)으로 연결한 지도입니다. 예: `파이어볼 ─소모한다→ 마나 30`, `고블린 전사 ─드랍한다→ 가죽 갑옷`. 지식 추출 시 자동으로 만들어지며, Chat 답변 시 관련 문서를 더 넓게 찾는 데에도 활용됩니다.

상단 4개의 서브탭은 각각 다음 상황에서 사용합니다.

### Knowledge — 그래프를 그림으로 보기

그래프를 선택하면 노드와 관계가 캔버스에 그려집니다. 관계 종류별로 색이 다르며, 범례를 클릭해 특정 관계만 필터링할 수 있습니다. 노드나 선을 클릭하면 상세 정보가 뜨고, 잘못 추출된 이름이나 관계는 **Edit**로 고치거나 **Delete**로 지울 수 있습니다.

> 여기서 고친 내용은 그래프에만 반영됩니다. 원본 위키 문서는 바뀌지 않으므로, 문서 내용 자체를 고치려면 Files 탭에서 문서를 편집하세요.

### Files — 위키 문서 읽고 편집하기

AI가 만든 위키 페이지들을 폴더별로 탐색합니다. 파일을 선택하면 내용 미리보기와 함께 이 문서가 연결된 그래프가 표시됩니다. **Edit** 버튼을 누르면 일반 문서 편집기처럼 바로 수정할 수 있는 에디터(수식 지원)가 열립니다.

### Graphs — 그래프 목록 관리하기

프로젝트에 어떤 그래프들이 있는지 관리합니다. 그래프 추가/이름 변경/삭제와, 각 그래프에서 사용할 관계 종류(그래프당 최대 4개) 편집이 가능합니다. 삭제 시 어떤 문서와의 연결이 끊어지는지 미리 알려줍니다.

### Entity — 같은 대상의 다른 이름 정리하기

자료마다 같은 대상을 다르게 부르는 경우("고블린 전사" / "Goblin Warrior" / "고블린전사")가 있습니다. Entity 탭의 **엔티티 사전**은 이런 이름들을 하나의 대표 이름 + 별칭 목록으로 묶어 관리합니다. 여기서 이름을 정리해 두면 이후 지식 추출 때 AI가 기존 이름을 재사용해 그래프가 깔끔하게 유지됩니다. 검색, 별칭 추가/삭제, 중복 엔티티 병합/분리가 가능합니다.

## 수정 요청 (Review) 탭

AI는 위키를 **마음대로 고치지 않습니다.** 기존 내용과 부딪히거나 판단이 애매한 경우 이 탭에 카드를 만들어 사용자에게 물어봅니다. 사이드바의 수정 요청 아이콘에 빨간 숫자가 떠 있다면 확인할 항목이 있다는 뜻입니다. 카드는 상황에 따라 4종류입니다.

### ① 문서 충돌 — "이미 있는 문서와 새 내용이 달라요"

예: 위키에 "파이어볼 쿨타임 8초"라는 문서가 있는데, 새로 넣은 패치노트에는 "쿨타임 6초"라고 되어 있는 경우. 기존 내용과 새 내용을 나란히 비교해 보여주며, 다음 중 선택합니다.

| 버튼 | 언제 누르나 | 결과 |
|---|---|---|
| **Approve** | 새 내용이 맞다 | 새 내용으로 교체되고 그래프에도 반영 |
| **Merge** | 둘을 섞어야 한다 | 새 내용 초안이 에디터로 열림 — 직접 다듬은 뒤 승인 |
| **Reject → Discard** | 새 내용이 틀렸다 | 기존 유지, 새 내용은 버림 |
| **Reject → Pending** | 지금 판단 못 하겠다 | 새 내용을 보류함에 넣어둠 (나중에 **pending** 서브탭에서 다시 처리) |
| **Reject → Counterexample** | 틀렸지만 기록할 가치가 있다 | "반례" 보관함에 따로 저장 |

### ② 엔티티 확인 — "이거 같은 대상인가요?"

새 자료에서 나온 이름이 기존 이름과 **비슷하지만 똑같지는 않을 때** 묻습니다. 예: 기존에 "고블린 전사"가 있는데 새 자료에 "고블린전사"가 등장.

- **같은 엔티티** — 별칭으로 등록되어 하나로 취급됩니다 (대부분 이걸 선택)
- **새 엔티티** — 실제로 다른 대상일 때
- **무시** — 판단을 미루고 일단 그대로 등록

### ③ 제안 — "자동으로 처리하지 못했어요"

관계를 그래프에 배정하지 못한 항목입니다. 예를 들어 그래프의 관계 종류 4개가 모두 차서 새 관계를 넣을 수 없을 때 "Create new graph" 버튼으로 새 그래프를 만들어 해결할 수 있습니다.

### ④ 스키마 변경 — "양식을 바꿔도 될까요?"

데이터 유형/질문 유형 양식 변경 제안을 승인하거나 거부합니다.

처리가 끝난 카드는 **해결 완료 항목 정리** 버튼으로 한 번에 치울 수 있습니다.

## 히스토리 & 원격 동기화

프로젝트의 모든 변경은 버전으로 기록됩니다(내부적으로 Git 사용 — Git을 몰라도 사용에 지장 없습니다). 덕분에 **무엇이 언제 바뀌었는지 확인하고, 실수한 변경을 되돌릴 수 있으며, 팀원과 같은 위키를 공유**할 수 있습니다.

### 히스토리 탭 — 변경 이력 보고 되돌리기

좌측에 변경 기록(커밋) 목록이 시간순으로 표시됩니다. 하나를 선택하면 그때 어떤 파일이 추가/수정/삭제됐는지, 내용이 어떻게 달라졌는지 전후 비교로 볼 수 있습니다.

되돌리기는 두 가지 단위로 가능합니다:

- **이 버전으로 복원** — 특정 **파일 하나**만 그 시점의 내용으로 되돌립니다. "이 문서만 어제 상태로" 할 때 사용.
- **이 커밋 되돌리기** — 그 기록에 포함된 **모든 변경**을 취소합니다. "이 자료 추출 전체가 잘못됐다" 할 때 사용. (되돌리는 중 다른 변경과 겹치면 어떤 파일이 문제인지 안내됩니다.)

### Sync to Remote — 저장하고 팀과 공유하기

앱 사용 중에는 변경 사항이 로컬 파일에만 기록되고 **자동으로 버전이 만들어지거나 서버에 올라가지 않습니다.** 작업이 일단락되면 직접 동기화하세요:

- 사이드바 하단 **Upload 버튼**을 누르거나,
- 앱을 닫을 때 나타나는 창에서 **Sync to Remote**를 선택합니다. (지금 안 올리려면 "Exit without Sync")

동기화하면 현재 상태 전체가 하나의 버전으로 기록되고, **팀원이 올린 변경을 먼저 받아온 뒤** 내 변경을 서버에 올립니다. 팀원과 내가 같은 문서의 같은 부분을 고쳤다면 충돌 해결 창이 떠서 어느 쪽을 남길지 선택할 수 있습니다.

> 원격 저장소 없이 만든 로컬 온리 프로젝트에서는 이 버튼이 표시되지 않습니다. 히스토리(이력·되돌리기)는 로컬 온리에서도 동일하게 동작합니다.

## 경우의 수 맵핑 (테스트 플랜)

게임 기능의 설명을 넣으면 **"어떤 조합을 테스트해야 하는가"를 AI와 함께 도출해 테스트케이스 문서까지 만들어 주는** QA 도구입니다. 사람이 놓치기 쉬운 조건 조합을 빠짐없이 찾는 것이 목적입니다.

**새 플랜**을 만들고 7단계를 순서대로 진행합니다. 각 단계에서 AI가 초안을 제시하면 사용자가 검토·수정하고 다음 단계로 넘어가는 방식입니다. "파이어볼 스킬"을 예로 들면:

1. **기능 정보 입력** — 기능을 아는 대로 자연어로 적습니다.
   > 예: "파이어볼은 적 1명을 대상으로 하는 마법 스킬. 대상에게 피해 + 주변 스플래시 피해. 마나 30 소모, 쿨타임 8초, 전투 중에만 사용 가능."
2. **추상화** — AI가 기능의 특징을 태그로 뽑아냅니다 (예: `단일 대상`, `마나 소모`, `쿨타임 있음`). 잘못 뽑힌 태그는 지우고 빠진 태그는 추가하세요. 위키에 이미 등록된 엔티티와 연결 제안이 뜨기도 합니다.
3. **테스트 축** — 테스트에서 바꿔볼 변수(축)와 그 값들을 AI가 추천합니다 (예: 축 "대상 상태" = 일반/사망 직전/무적). 해당 없는 축은 끄고, 빨간색으로 표시된 값은 특히 위험한 값이라는 뜻입니다.
4. **조합 생성** — 축의 값들을 조합해 테스트할 경우의 수를 만듭니다. 모든 조합을 다 만들면 수가 폭발하므로, **페어와이즈 방식**(임의의 두 값 짝이 최소 한 번씩은 등장하도록 하는 검증된 축소 기법)으로 실용적인 개수만 생성합니다.
5. **불가능 조합 제거** — "잠수 중 + 스킬 사용" 같은 게임상 불가능한 조합을 걸러냅니다. 먼저 등록된 룰이 적용되고, 나머지는 AI가 판정합니다. 판정이 틀렸으면 **뒤집기**로 수정하세요. 여기서 만든 룰은 **다른 플랜에도 공유**되므로 플랜이 쌓일수록 판정이 정확해집니다.
6. **리스크 우선순위** — 남은 조합에 위험 등급을 매깁니다. 재화 손실·진행 불가·중복 지급으로 이어질 수 있는 조합이 High, 연출 수준 문제는 Low입니다. 등급이 높은 것부터 테스트하면 됩니다.
7. **QA 리뷰** — 각 조합이 목적·사전 조건·절차·기대 결과를 갖춘 테스트케이스로 완성됩니다. 케이스를 편집/제외하거나 수동 케이스를 추가한 뒤 **최종 확정 & 내보내기**를 누르면 파일로 저장됩니다.

진행 상황은 플랜별로 저장되므로 중간에 끊고 나중에 이어서 작업해도 됩니다.

## 페르소나 시나리오

**"이런 성향의 유저라면 이 기능을 어떻게 쓸까?"를 AI로 시뮬레이션**해 테스트 시나리오를 만드는 도구입니다. 개발자나 QA가 예상하는 정석 플레이 말고, 실제 유저처럼 비정상적·극단적으로 쓰는 경우를 미리 찾아볼 수 있습니다.

두 개의 서브탭으로 구성됩니다.

1. **페르소나 관리** — 유저 유형을 등록합니다. 이름, 설명, 특성(쉼표로 구분)을 입력하세요.
   > 예: 이름 "숙련 유저" / 설명 "빠른 입력을 많이 하는 숙련 유저" / 특성 "연타, 동시 입력"
   >
   > 그 외에도 "초보 유저(메뉴를 헤맴)", "악용 시도 유저(버그를 노림)" 같은 페르소나를 만들어 두면 유용합니다.
2. **시나리오** — 페르소나를 선택하고, 테스트할 대상 기능(예: "파이어볼 스킬 전투 루프")과 생성 개수를 입력한 뒤 **시나리오 생성**을 누릅니다. 해당 페르소나가 그 기능을 어떻게 플레이할지에 대한 시나리오가 만들어집니다.
   > 예: 숙련 유저 페르소나라면 "쿨타임이 끝나는 프레임에 맞춰 파이어볼을 연타하면서 다른 스킬을 동시 입력한다" 같은 시나리오가 나옵니다.

생성된 시나리오는 개별 삭제하거나 **내보내기**로 파일로 저장할 수 있습니다.

## 브라우저 확장 — LLM Wiki Clipper

웹서핑 중 보고 있는 페이지를 원클릭으로 위키에 추출하는 Chrome 확장입니다.

### 설치

1. Chrome에서 `chrome://extensions` 접속 → **개발자 모드** 활성화
2. **압축해제된 확장 프로그램을 로드** → 이 저장소의 `extension/` 폴더 선택

### 사용

1. LLM Wiki 앱을 실행하고 프로젝트를 열어 둡니다. 앱이 로컬 clip 서버(`127.0.0.1:19827`)를 구동하며, 사이드바 하단 상태 점(녹색 = 실행 중)과 설정 > 정보에서 상태를 확인할 수 있습니다.
2. 클리핑할 페이지에서 확장 아이콘 클릭 → 본문이 자동 추출(Readability)되어 markdown 미리보기가 표시됩니다. 제목을 수정하고 대상 프로젝트를 드롭다운에서 선택한 뒤 **Clip**을 누릅니다.
3. 앱이 3초 이내에 클립을 감지해 자동으로 지식 추출 큐에 추가합니다 (LLM 설정이 되어 있을 때).

## 프로젝트 폴더 구조

프로젝트는 일반 폴더 + git 저장소이며, 모든 데이터가 파일로 저장되어 그대로 백업/공유할 수 있습니다.

```
my-wiki/
├── db/                  # 위키 페이지 (markdown) — 검색·답변의 대상
├── raw/sources/         # 추출 원본 자료 (파일/URL/웹클립)
├── pending/             # 보류된 초안 (+ _proposals/: 충돌 초안)
├── counterexamples/     # 반례로 분류한 초안
├── question_types/      # 질문 유형 정의 (YAML) — 프로젝트 공유
├── data_types/          # 데이터 유형 정의 (YAML) — 프로젝트 공유
├── graph.json           # 지식 그래프 스냅샷 (Sync 시 갱신)
└── .llm-wiki/           # 내부 상태 (graph.sqlite, entity-dict.json,
                         #   graph-policy.json, page-graph-index.json 등)
```

## 질문 유형 / 데이터 유형 커스터마이징

### 데이터 유형 (`data_types/*.yaml`)

지식 추출 시 구조화 추출에 사용됩니다.

```yaml
name: "패치노트"
fields:
  version: "패치 버전 번호"
  release_date: "패치 적용 날짜"
  balance_changes: "수치 변경 내용"
  bug_fixes: "버그 수정 목록"
```

### 질문 유형 (`question_types/*.yaml` 또는 `.md`)

Chat에서 답변 형식을 강제할 때 사용됩니다.

```yaml
name: "버그 분석"
description: "버그 원인과 해결책을 묻는 질문"
fields:
  cause: "원인 설명"
  solution: "해결 방법"
prompt_template: |
  Respond with each field listed above clearly labeled.
```

프로젝트의 해당 폴더에 파일을 추가하면 즉시 사용할 수 있습니다. 개인 전용 override는 `.llm-wiki/question-types/`, `.llm-wiki/data-types/`에 둘 수 있습니다.
